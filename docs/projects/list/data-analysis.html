Overview

Every retail business generates transaction logs — thousands of rows tracking what sold, when, and where. The problem is rarely a lack of data. It’s knowing what to look at first.1

This project takes a synthetic retail dataset and works through a structured EDA process: cleaning, understanding the shape, then asking progressively sharper questions until a few genuinely useful patterns emerge.

Dataset

The dataset is a synthetic retail sales log covering 12 months across four product categories.

Column Type Description
date datetime Transaction date
category string Product category
region string Irish region
units_sold int Units per transaction
revenue float Revenue in euros (€)

Setup & Data Loading

Code 
import pandas as pd
import plotly.express as px
import plotly.graph_objects as go
import numpy as np

np.random.seed(42)

dates = pd.date_range("2024-01-01", "2024-12-31", freq="D")
categories = ["Electronics", "Apparel", "Home & Garden", "Sports"]
regions = ["Dublin", "Cork", "Galway", "Limerick"]

rows = []
for date in dates:
    for _ in range(np.random.randint(3, 8)):
        cat = np.random.choice(categories)
        base = {"Electronics": 180, "Apparel": 55, "Home & Garden": 90, "Sports": 70}[cat]
        seasonal = 1 + 0.3 * np.sin((date.month - 1) * np.pi / 6)
        rows.append({
            "date": date,
            "category": cat,
            "region": np.random.choice(regions),
            "units_sold": int(np.random.poisson(seasonal * 5)),
            "revenue": round(base * seasonal * np.random.uniform(0.85, 1.15) * np.random.poisson(seasonal * 5), 2)
        })

df = pd.DataFrame(rows)
df["month"] = df["date"].dt.to_period("M").astype(str)
df["month_num"] = df["date"].dt.month

print(f"Rows: {len(df):,} | Columns: {df.shape[1]}")

Revenue Formula

The synthetic revenue for each transaction is modelled as:

\[R = B \cdot S \cdot \epsilon \cdot U\]

Where:

  • \(B\) = base price per category
  • \(S\) = seasonal multiplier \(\left(1 + 0.3\sin\left(\frac{(m-1)\pi}{6}\right)\right)\)
  • \(\epsilon\) = random noise \(\sim \text{Uniform}(0.85, 1.15)\)
  • \(U\) = units sold \(\sim \text{Poisson}(5S)\)

Monthly Revenue Trend

Code 
monthly = df.groupby("month")["revenue"].sum().reset_index()
monthly.columns = ["month", "total_revenue"]

fig = px.line(
    monthly, x="month", y="total_revenue",
    title="Monthly Revenue — Full Year",
    labels={"month": "", "total_revenue": "Revenue (€)"},
    markers=True,
    color_discrete_sequence=["#7fff6e"]
)
fig.update_layout(
    plot_bgcolor="#14141f", paper_bgcolor="#14141f",
    font_color="#9898b0", title_font_color="#ffffff",
    xaxis=dict(gridcolor="#252535", tickangle=-30),
    yaxis=dict(gridcolor="#252535"),
)
fig.update_traces(line_width=2.5)
fig.show()

Monthly revenue across the full year, showing a clear seasonal peak in Q4.

Key Takeaways

NoteWhat this analysis found
  • Revenue follows a clear seasonal arc peaking in Q4 — promotional spend should be front-loaded into Q3 to capture the ramp-up.
  • Electronics generates disproportionate revenue relative to transaction count — protecting margin here matters more than volume discounts.
  • Regional revenue is broadly distributed, so a Dublin-only marketing focus would miss a substantial share of the market.
TipNext steps

A natural follow-up is forecasting: does the Q4 peak come from higher basket sizes, or more transactions? That’s the next query to write.

Reflection

This project reinforced something I keep running into — the most useful output of an EDA isn’t the visualisations themselves, it’s the shortlist of questions they raise. Clean data still takes time, even when it’s synthetic. There were real decisions to make about aggregation level, date formatting, and how to handle multi-category breakdowns without obscuring the signal.

$$R = B \cdot S \cdot \epsilon \cdot U$$
