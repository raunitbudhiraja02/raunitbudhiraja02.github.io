Reflections

On This Module

When this module started I thought an ePortfolio was just a fancier CV — a place to dump projects and call it done. That framing turned out to be wrong in a useful way.

Building this site made clear that communicating technical work is its own skill, completely separate from doing the work. Writing about why I made the choices I made, and what the output means to someone who isn’t going to read my code — that takes a different kind of effort, and it’s one I underestimated at the start.

Skills Gained

Git and Version Control

I’d used GitHub before to store code, but that’s not really using it — that’s using it as a USB drive with version history. Understanding the actual workflow of add, commit, push — and why each step matters — changed how I think about managing any project, technical or not. The idea that every commit is a recoverable snapshot, and that commit messages are a form of documentation, is something I’ll carry forward.

TipKey insight on Git

A clean commit history tells a story. Future you — or a collaborator — will thank present you for writing “fix null handling in revenue column” instead of “update”.

Quarto

Quarto was a genuine surprise. I expected it to behave like Jupyter Notebooks — functional but clunky to publish. Instead it’s genuinely elegant: write in Markdown, embed live code, render to a clean website. The separation between content and presentation is well-designed, and the YAML-driven configuration means you can change the entire look of a site by editing a few lines.

Python for Analytics

The progression across the three coding projects this year — EDA, then classification, then dashboards — forced me to think about data at different levels of abstraction. EDA is exploratory and inductive. Machine learning is more formal, with explicit evaluation criteria. Dashboards require you to think about the reader’s questions, not your own. They’re related skills but they feel quite different in practice.

On Data Ethics

Completing the Kubicle courses on GDPR and data ethics ran alongside building the machine learning project, and the timing mattered. Reading about algorithmic accountability while actually building a classifier made the abstract problems feel concrete.

The churn prediction model — if a real company deployed it, you’d immediately have questions worth asking: are certain demographic groups being systematically underserved by the model? Does error rate differ across segments? Is “reducing churn” actually the right thing to optimise for, or is it a proxy for something else?

I don’t think these questions have clean answers. But I think asking them consistently — treating ethics as part of the analytical workflow, not a disclaimer at the end — is the right habit to build early.1

An Inspiring Talk on Data Communication

One thing that shaped how I approached this portfolio was watching this talk on communicating data to non-technical audiences:

Hans Rosling, “The best stats you’ve ever seen” — TED, 2006.

The lesson I took from it: the most powerful visualisation is one that makes a non-expert say “I never thought about it that way.” That’s the bar I was trying to hit with the dashboard project.

What I’d Do Differently

Start the presentation layer earlier. The analysis for each project took a few hours. Getting the Quarto rendering right, debugging the SCSS, fixing the navbar — easily double that. Next time I’d treat the site itself as part of the project from day one, not an afterthought.

Use real data. I relied on synthetic datasets throughout, which was pragmatic — real data has privacy considerations and messy cleaning problems — but it also meant the uncertainty was fake. Real data is messier and more interesting.

Write the reflection first. The reflections are the hardest part of this portfolio, not because I don’t have things to say, but because saying them clearly — without understating what I learned or overclaiming insight I don’t actually have — requires more care than the technical sections. Starting with “what do I actually think I learned?” before writing the project pages would have made both better.

Looking Forward

These skills — Python, Git, Quarto, data communication — aren’t interesting to me as credentials. They’re interesting as tools that let me answer questions I actually care about.

In the short term I’m looking for a data analyst or junior data science role where analytics sits close to product decisions. In the longer term I’m interested in how organisations build data cultures — not just how to run models, but how to make data-informed thinking the default for a team.

This portfolio is the beginning of that record, not the end of it. The projects will get more complex, the code will get cleaner, and — I hope — the questions will get sharper.

NoteConnecting back to the module

BAA1104 gave me the infrastructure to present work professionally. The ePortfolio, Git workflow, and Quarto setup aren’t just for this assignment — they’re the scaffolding I’ll use for everything I build going forward.
